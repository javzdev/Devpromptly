1. QUE ES ESTE PROYECTO
================================================================================

Gpromts es una plataforma web full-stack para descubrir, crear y compartir
prompts de inteligencia artificial. Funciona como una red social especializada
donde los usuarios pueden:

- Publicar prompts para herramientas de IA (ChatGPT, Claude, Midjourney, etc.)
- Buscar y filtrar prompts por categoria o herramienta
- Calificar prompts con estrellas (1-5)
- Guardar prompts favoritos
- Reportar contenido inapropiado
- Leer y escribir articulos en el blog
- Descubrir foros y comunidades de IA
- Ver un directorio de herramientas de IA

El proyecto tiene un sistema completo de moderacion con roles de admin,
moderador y usuario normal. Todo el codigo esta orientado a produccion
con seguridad enterprise-level.

4. VARIABLES DE ENTORNO

Archivo de plantilla: .env.example

Las variables necesarias para correr el proyecto son:

--- BASE DE DATOS ---
MONGODB_URI=mongodb://localhost:27017/gpromts
  -> URI de conexion a MongoDB. En produccion usar MongoDB Atlas.

--- AUTENTICACION JWT ---
JWT_SECRET=tu_clave_secreta_aqui
  -> Clave para firmar access tokens (15 minutos de vida)
JWT_REFRESH_SECRET=tu_otra_clave_secreta
  -> Clave para firmar refresh tokens (7 dias de vida)
JWT_EXPIRES_IN=15m
  -> Tiempo de expiracion del access token
JWT_REFRESH_EXPIRES_IN=7d
  -> Tiempo de expiracion del refresh token

--- SERVIDOR ---
PORT=5000
  -> Puerto donde corre el servidor Express
NODE_ENV=development
  -> Entorno: development o production

--- SESIONES ---
SESSION_SECRET=clave_para_sesiones
  -> Clave para firmar cookies de sesion

--- SEGURIDAD ---
ALLOWED_ORIGINS=http://localhost:3000,http://localhost:3001
  -> Origenes permitidos por CORS (separados por coma)
BCRYPT_ROUNDS=12
  -> Rondas de hashing para contrasenas (mas = mas lento = mas seguro)

--- CLIENTE REACT ---
REACT_APP_API_URL=http://localhost:5000
  -> URL base de la API (usada por el frontend)

--- CLOUDFLARE (produccion) ---
CLOUDFLARE_API_TOKEN=token_de_cloudflare
CLOUDFLARE_ACCOUNT_ID=id_de_cuenta

5. BASE DE DATOS - MODELOS
================================================================================

La base de datos es MongoDB. Todos los modelos usan Mongoose.

--- MODELO: User ---
Representa a un usuario registrado en la plataforma.

Campos principales:
  username         String, unico, requerido - nombre de usuario
  email            String, unico, requerido - correo electronico
  password         String, hasheado con bcrypt
  avatar           String - URL del avatar
  role             Enum: 'user', 'moderator', 'admin'
  isActive         Boolean - cuenta activa o suspendida
  favorites        Array de IDs de prompts guardados

Autenticacion:
  refreshToken     String - token de refresh activo
  loginAttempts    Number - intentos fallidos de login
  lockUntil        Date - cuenta bloqueada hasta esta fecha

Verificacion de email:
  emailVerified              Boolean
  emailVerificationToken     String
  emailVerificationExpires   Date

Restablecimiento de contrasena:
  passwordResetToken         String
  passwordResetExpires       Date

Estadisticas:
  stats.promptsCreated   Number
  stats.totalRatings     Number
  stats.averageRating    Number

Preferencias:
  preferences.showNSFW   Boolean - mostrar contenido adulto
  preferences.language   String

Soft delete:
  deletedAt    Date - si tiene valor, el usuario esta "eliminado"

Metodos del modelo:
  comparePassword(password)  - Compara contrasena con hash
  softDelete()               - Elimina suavemente (preserva datos)
  restore()                  - Restaura cuenta eliminada
  updateStats()              - Recalcula estadisticas

Indices: username, email, stats.promptsCreated, createdAt

---

--- MODELO: Prompt ---
Representa un prompt creado por un usuario.

Campos principales:
  title        String, requerido - titulo del prompt
  description  String - descripcion corta
  prompt       String, requerido - el texto del prompt en si
  category     Enum - categoria del prompt:
               text-generation, image-generation, code-generation,
               audio-generation, video-generation, data-analysis,
               translation, other
  aiTool       String - herramienta AI para la que fue creado
  tags         Array de strings - etiquetas
  isNSFW       Boolean - contenido adulto
  author       Referencia a User

Estado de moderacion:
  status         Enum: 'pending', 'approved', 'rejected'
  moderatedBy    Referencia a User (admin/moderador)
  moderatedAt    Date
  rejectionReason  String

Sistema de calificaciones:
  ratings.average      Number - promedio de calificaciones
  ratings.count        Number - total de calificaciones
  ratings.distribution  Object - {1: n, 2: n, 3: n, 4: n, 5: n}
  ratings.userRatings  Array - {user: ID, rating: 1-5} (evita duplicados)

Engagement:
  favorites    Number - cuantas veces fue guardado como favorito
  views        Number - cuantas veces fue visto
  featured     Boolean - destacado por admins

Reportes:
  reportedCount    Number
  lastReportedAt   Date

Soft delete:
  deletedAt    Date

Metodos:
  calculateAverageRating()   - Recalcula el promedio
  addRating(userId, rating)  - Agrega/actualiza calificacion (atomico)

---

--- MODELO: BlogPost ---
Articulos del blog de la plataforma.

Campos principales:
  title        String - titulo del articulo
  slug         String, unico - URL amigable (auto-generado)
  excerpt      String - resumen
  content      String - contenido completo (HTML/Markdown)
  category     Enum: guide, tutorial, ranking, news, photo, review, opinion
  coverImage   String - URL de imagen de portada
  images       Array de URLs
  author       Referencia a User

Menciones (links a contenido de la plataforma):
  mentions     Array de {type: 'tool'|'prompt'|'community', id, name, url}

Engagement:
  views        Number
  likes        Array de IDs de usuarios
  comments     Array de {author, content, createdAt, likes}

Publicacion:
  published    Boolean
  featured     Boolean

Metadata (auto-calculada):
  readTime     Number - tiempo estimado de lectura en minutos
  tags         Array de strings

Hooks pre-save:
  - Genera slug automaticamente desde el titulo
  - Calcula readTime basado en palabras del contenido

--- MODELO: AuditLog ---
Registro de todas las acciones administrativas.

Acciones registradas:

  forum_delete, tool_create, tool_update, tool_delete

Campos:
  performedBy          ID del admin que realizo la accion
  performedByUsername  String (desnormalizado para historial)
  performedByRole      String
  action               Tipo de accion
  targetType           Que tipo de objeto fue afectado
  targetId             ID del objeto afectado
  details              Object - datos del cambio
  reason               String - razon opcional
  ipAddress            String
  userAgent            String
  success              Boolean

Metodos estaticos:
  createLog(data)              - Crear un log
  getRecentActions(limit)      - Ultimas acciones
  getActionsByAdmin(adminId)   - Acciones de un admin especifico
  getActionsOnTarget(type, id) - Historial de un objeto

6. BACKEND - SERVIDOR
================================================================================

Archivo principal: server/indexEnhanced.js

El servidor Express se configura con las siguientes capas (en orden):

1. CONFIGURACION DE CONFIANZA
   app.set('trust proxy', 1)
   -> Necesario cuando hay un load balancer o proxy (como Nginx/Cloudflare)

2. MIDDLEWARES DE SEGURIDAD
   - helmet()              -> Headers HTTP de seguridad
   - Content-Security-Policy (CSP) con directivas estrictas
   - X-Content-Type-Options: nosniff
   - X-Frame-Options: DENY (evita clickjacking)
   - Permissions-Policy: sin acceso a geoloc, microfono, camara
   - Referrer-Policy: strict-origin-when-cross-origin

3. CORS
   Permite peticiones desde los origenes configurados en ALLOWED_ORIGINS.
   Por defecto: localhost:3000 y localhost:3001

4. COMPRESION Y PARSEO
   - compression()         -> Comprime respuestas HTTP con gzip
   - express.json()        -> Parsea body JSON (limite 10kb)
   - express.urlencoded()  -> Parsea forms (limite 10kb)
   - cookieParser()        -> Parsea cookies

5. LOGGING
   - morgan('combined')    -> Log de peticiones en produccion
   - morgan('dev')         -> Log colorido en desarrollo

6. RATE LIMITING
   - /api/*:    Limite general de peticiones
   - /api/auth/*: Limite estricto en endpoints de autenticacion

7. SANITIZACION
   - mongoSanitize()       -> Previene inyeccion NoSQL
   - xss()                 -> Limpia inputs con XSS
   - hpp()                 -> Evita HTTP Parameter Pollution

8. SESIONES Y CSRF
   - sessionMiddleware     -> Sesiones con Redis opcional
   - generateCSRFToken     -> Genera token CSRF para el cliente

9. MONTAJE DE RUTAS
   app.use('/api/auth',    authRoutes)
   app.use('/api/prompts', promptRoutes)
   app.use('/api/users',   userRoutes)
   app.use('/api/admin',   adminRoutes)
   app.use('/api/forums',  forumRoutes)
   app.use('/api/tools',   aiToolRoutes)
   app.use('/api/blog',    blogRoutes)

10. CONEXION A MONGODB
    mongoose.connect(MONGODB_URI) con opciones de reconnect

  - Verifica la firma con JWT_SECRET
  - Busca el usuario en MongoDB
  - Verifica que la cuenta este activa
  - Agrega req.user al request

authenticateAdmin
  - Llama authenticateToken primero
  - Verifica que role sea 'admin' o 'moderator'
  - Error 403 si no tiene permisos

authenticateSuperAdmin
  - Solo permite role 'admin'
  - Para operaciones criticas como gestionar moderadores

optionalAuth
  - Como authenticateToken pero no falla si no hay token
  - Usado en rutas publicas que muestran contenido diferente si estas logueado

refreshTokenMiddleware
  - Lee el refresh token de la cookie httpOnly
  - Genera un nuevo access token si el refresh es valido
  - Invalida el refresh token anterior (rotacion de tokens)

checkAccountStatus
  - Verifica que isActive sea true
  - Retorna 403 si la cuenta esta suspendida

requireEmailVerification
  - Verifica que emailVerified sea true
  - Para funciones que requieren email verificado

---

--- SESIONES Y CSRF (session.js) ---

sessionMiddleware
  - Configura express-session
  - En produccion: usa Redis como store
  - En desarrollo: usa store en memoria
  - Cookies: httpOnly, secure en produccion, sameSite: 'lax'
  - Expiracion: 24 horas

generateCSRFToken
  - Genera token aleatorio de 32 bytes (hex)
  - Lo guarda en la sesion del usuario
  - El cliente debe incluirlo en el header X-CSRF-Token

validateCSRFToken
  - Verifica el token CSRF en peticiones POST/PUT/PATCH/DELETE
  - Compara el header X-CSRF-Token con el de la sesion
  - Error 403 si no coinciden

--- RATE LIMITING (rateLimiter.js) ---

apiRateLimiter (aplicado a /api/*)
  - 100 peticiones por ventana de 15 minutos por IP
  - Headers informativos: RateLimit-Limit, RateLimit-Remaining

authRateLimiter (aplicado a /api/auth/*)
  - 20 peticiones por ventana de 15 minutos por IP
  - Mas estricto para prevenir fuerza bruta

passwordResetRateLimiter
  - 5 intentos por hora por IP
  - Previene abuso del sistema de reset

searchRateLimiter
  - 30 busquedas por minuto por IP
  - Previene scraping masivo

En produccion: usa Redis para distribuir el conteo entre multiples instancias.
En desarrollo: usa store en memoria.

12. SERVICIO API DEL CLIENTE
================================================================================

Archivo: client/src/services/api.ts (572 lineas)

Este archivo centraliza TODAS las llamadas HTTP al backend.

INSTANCIA DE AXIOS:
  - Base URL: REACT_APP_API_URL o http://localhost:5000
  - withCredentials: true (envia cookies, necesario para refresh token)
  - timeout: 10000 (10 segundos)

REINTENTOS AUTOMATICOS (axios-retry):
  - 3 reintentos en errores de red
  - Backoff exponencial: espera mas entre cada reintento
  - No reintenta en errores 4xx (son errores del cliente)

CACHE SIMPLE:
  - TTL: 5 minutos
  - Solo para peticiones GET
  - Se limpia automaticamente en mutaciones (POST/PUT/DELETE)
  - Implementado en memoria (Map)

INTERCEPTOR DE REQUEST:
  - Agrega automaticamente el header X-CSRF-Token a peticiones
    que modifican datos (POST, PUT, PATCH, DELETE)
  - El token CSRF se obtiene de memoria (secureStorage)

INTERCEPTOR DE RESPONSE:
  - En error 401: intenta renovar el token automaticamente
  - Cola de requests: si hay token refrescandose, encola los requests fallidos
  - Una vez renovado el token, reintenta todos los requests en cola
  - En error 403: muestra mensaje de "sin permisos"
  - En error 429: muestra "demasiadas peticiones, espera un momento"
  - En error de red: muestra "error de conexion"

ALMACENAMIENTO SEGURO (secureStorage):
  - El token CSRF se guarda en MEMORIA (variable JS), no en localStorage
  - Esto protege contra ataques XSS que roben tokens de localStorage
  - El access token tambien vive en memoria via AuthContext

13. SISTEMA DE AUTENTICACION
================================================================================

El sistema usa JWT con dos tokens:

ACCESS TOKEN:
  - Vida corta: 15 minutos
  - Se guarda en MEMORIA (variable JS en AuthContext)
  - Se envia en el header: Authorization: Bearer <token>
  - Firmado con JWT_SECRET
  - Contiene: { userId, role }

REFRESH TOKEN:
  - Vida larga: 7 dias
  - Se guarda en cookie httpOnly, secure, sameSite
  - El JavaScript NO puede leerlo (proteccion XSS)
  - Firmado con JWT_REFRESH_SECRET
  - Contiene: { userId } + referencia en DB

FLUJO DE LOGIN:
  1. Usuario envia email + password
  2. Servidor verifica credenciales con bcrypt
  3. Si son correctas, genera access token + refresh token
  4. Refresh token: se guarda en DB (campo refreshToken del User)
     y se envia como cookie httpOnly
  5. Access token: se devuelve en el body JSON
  6. Cliente guarda access token en memoria
  7. Cliente usa access token en cada request

FLUJO DE RENOVACION (token refresh):
  1. Request con access token expirado -> respuesta 401
  2. El interceptor de axios detecta el 401
  3. Hace POST /api/auth/refresh (enviar cookie automaticamente)
  4. Servidor verifica la cookie con el refresh token
  5. Si es valido, genera nuevo access token (y nuevo refresh token)
  6. Viejo refresh token se invalida en DB
  7. Nuevo access token se guarda en memoria
  8. Se reintenta el request original con el nuevo token

FLUJO DE LOGOUT:
  1. POST /api/auth/logout
  2. Servidor elimina refreshToken del usuario en DB
  3. Servidor limpia la cookie del refresh token
  4. Cliente limpia el access token de memoria
  5. AuthContext resetea el estado user a null

PROTECCION CONTRA FUERZA BRUTA:
  - Despues de N intentos fallidos: cuenta se bloquea por X minutos
  - Rate limiter: max 20 peticiones a /api/auth/* cada 15 minutos
  - El numero de loginAttempts se guarda en el modelo User

  - Easing: cubic-bezier(0.22, 1, 0.36, 1) (ease-out)
  - Duracion tipica: 150-200ms

REGLAS DEL SISTEMA:
  - Un solo color de acento (azul). Ambar solo para alertas.
  - Superficies con tinte azulado para coherencia.
  - Sidebar mismo color que contenido (no gris separado).
  - Bordes siempre en tono del acento.

21. COMO CORRER EL PROYECTO LOCALMENTE
================================================================================

PREREQUISITOS:
  - Node.js 18+ instalado
  - MongoDB instalado y corriendo (o cuenta en MongoDB Atlas)
  - Git instalado

PASOS:

1. CLONAR / UBICARSE EN EL PROYECTO
   cd c:\Users\josec\Desktop\Gpromts

2. INSTALAR DEPENDENCIAS
   npm install          # instala dependencias del raiz (servidor)
   cd client
   npm install          # instala dependencias del cliente
   cd ..

3. CONFIGURAR VARIABLES DE ENTORNO
   Copiar .env.example a .env.development
   cp .env.example .env.development

   Editar .env.development con tus valores:
   MONGODB_URI=mongodb://localhost:27017/gpromts
   JWT_SECRET=una_clave_secreta_larga_y_random
   JWT_REFRESH_SECRET=otra_clave_secreta_diferente
   PORT=5000
   NODE_ENV=development
   SESSION_SECRET=clave_para_sesiones

4. CREAR USUARIO ADMIN INICIAL (opcional)
   cd server
   node utils/seedAdmin.js
   cd ..

   (Este script crea un usuario admin para acceder al panel)

5. CORRER EN DESARROLLO
   npm run dev

   Esto corre en paralelo:
   - Servidor en http://localhost:5000
   - Cliente en http://localhost:3000

6. ACCEDER A LA APLICACION
   Abrir http://localhost:3000 en el navegador

SCRIPTS DISPONIBLES (package.json raiz):
  npm run dev        - Servidor + cliente en paralelo (desarrollo)
  npm run server     - Solo el servidor con nodemon
  npm run client     - Solo el cliente React
  npm run build      - Compilar cliente para produccion
  npm start          - Servidor en modo produccion
  npm run audit      - Auditoria de seguridad de dependencias
